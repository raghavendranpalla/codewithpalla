Copilot CLI for everyday tasks: where its context comes from, and what a task really costs

By Palla Raghavendran · 11 July 2026 · 10 min read

Our team has been moving routine engineering chores — failure triage, test-design drafts, small refactors, release-note summaries — from the IDE chat window to GitHub Copilot CLI, the terminal agent. It's a different way of working: you give it a task, it plans, reads files, edits and runs commands in a loop. And it raises two questions every team asks in week one:

  • "What does the agent already know before I type my first word?" (and where does that knowledge come from — cache, instruction files, or memory?)
  • "What is each task doing to our AI Credit pool?" — because since the June 2026 move to usage-based AI Credits, honestly, almost nobody on a team can tell you their balance.

This article answers both, with the exact commands to see it for yourself.

What the agent knows at turn zero: run /context

Copilot CLI ships a slash command that most people discover far too late. Type /context in a session and it draws a map of the model's context window — who is occupying it and how much. It looks roughly like this (numbers illustrative):

Model: base agent model · 34k / 200k tokens used (17%) System prompt ~4k ██ Custom instructions ~2k █ ← your AGENTS.md etc. System tools ~11k █████ MCP tools ~9k ████ ← Playwright MCP, etc. Messages ~8k ████ ← the conversation so far Free space 166k

So before you say a word, the window already carries four layers. Understanding which layer is which tells you what's cheap, what's yours to control, and what the agent "remembers" on its own.

Layer 1 — preloaded and (mostly) cached: the system prompt and tool definitions

The system prompt and the built-in tool definitions are GitHub's, not yours — they load in every session, identical every time. Add MCP servers and their tool definitions join too. Identical-every-time is exactly what prompt caching is for: context the model has already processed is re-billed as cached input, typically at about 10% of the fresh-input price. Two practical consequences your team should know:

  • Don't switch models mid-session. A different model cannot reuse another model's cache, so the whole context is re-processed at full price. Pick the model, then start the task.
  • Caches expire on idle — roughly 24 hours for OpenAI models and about 1 hour for most others. A session you abandon over lunch may pay a re-warm on return.

Layer 2 — instruction files: the part you write once and pay for always

The "Custom instructions" line in /context is the sum of every instruction file the CLI found:

  • AGENTS.md in the repository — the agent-facing rulebook (build commands, conventions, definition of done).
  • .github/copilot-instructions.md — repository instructions, shared with Copilot's other surfaces (chat, code review, cloud agent).
  • ~/.copilot/instructions/*.instructions.md — your personal, global instructions that apply in every repository (coding style, "always answer with runnable commands", language preferences).

These files are injected into every session. That's their power — the agent starts already knowing your stack — and their tax: a bloated instructions file is input tokens on every request from every user, cached or not. My rule from the credits article stands: one screen of the things the agent gets wrong without it, nothing more.

Layer 3 — Copilot Memory: what the agent learned by itself

This is the layer teams are most surprised by. Copilot Memory (used by Copilot CLI, the cloud coding agent and Copilot code review) stores small facts the agent validated while working with you — "tests run with npm run test:api", "this repo uses YAML-driven scenarios" — and retrieves the relevant ones into context on later tasks. The mechanics that matter:

  • Repo-scoped, permission-gated. Memories are created only from actions in a repository by people with write access, and only used in tasks on that same repository. Your conventions don't leak across projects.
  • Self-cleaning. A memory that goes unused for 28 days is deleted automatically; using one resets its clock.
  • You control it per surface: in the CLI, /memory on, /memory off, and /memory show to see what it currently believes. The setting persists across sessions.

Think of the difference this way: instruction files are what you decided to tell the agent; memory is what it noticed. Both arrive as context tokens. Memory's advantage is that it's small, targeted and retrieved only when relevant — you don't pay for a 500-line wiki on every request.

"Nobody knows how many credits we've used" — here's how to actually see it

The old "300 premium requests" intuition died in June; a task now costs whatever its tokens cost, which is why nobody's gut feeling works anymore. The visibility tools exist — they're just not where people look:

  • /usage (in a CLI session) — session statistics including the AI Credits burned so far in this session. Make "check /usage before you close" a team habit for the first month.
  • /limits — view or set a per-session credit cap (added July 2026). The session tracks everything — model calls, subagents, background compaction — and stops at your ceiling instead of surprising you. Give juniors a default limit while they learn what tasks cost.
  • The org dashboard (billing → AI usage) — where the pooled Enterprise picture lives: consumption by user, by model, by day. Admins can set budgets per cost-center or per user.

Habit for adopters: /context when you start, /usage before you close, /limits if you're experimenting.

So how many tasks does an Enterprise seat actually buy?

A Copilot Enterprise seat includes 3,900 AI Credits per user per month (Business: 1,900), pooled across the org, no rollover. A credit is $0.01 of model usage — so the honest answer to "how many tasks?" is "it depends on the task's tokens". From our own adoption, with the base model and a lean setup, tasks cluster into three sizes (illustrative averages, not GitHub's numbers — measure your own with /usage):

  • Small (≈3–15 credits): explain a failure, draft a test-data table, a one-file edit with review. An Enterprise seat covers hundreds per month.
  • Medium (≈30–80 credits): a scoped agent task — "add validation to these three endpoints following the pattern in orders.ts, run the tests". Roughly 50–130 per seat per month if that's all you did.
  • Large (≈200–500+ credits): long multi-hour agentic sessions, frontier models, big-context refactors. These are a few per week per seat — budget them like you'd budget a specialist's afternoon.

The levers that move a task between buckets, in order of impact: model choice (frontier vs base is often the single biggest multiplier), conversation length (every turn re-sends history — one task per session), and context hygiene (compaction kicks in automatically near 80% of the window, but /compact before a new phase of work keeps you in cheap territory).

The adoption playbook we settled on

  1. One AGENTS.md per repo, one screen long. Review it monthly like code.
  2. /memory on for the repos you work in daily; /memory show in retros so the team sees what it learned.
  3. Base model by default; frontier models are an escalation, not a habit.
  4. One task per session. Finishing a task and starting an unrelated one in the same thread is paying to re-send the first task forever.
  5. /limits caps for new users; /usage before closing any long session.
  6. Admins: watch billing → AI usage weekly, and set per-user budgets before the first month-end surprise.
